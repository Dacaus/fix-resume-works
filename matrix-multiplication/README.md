# 矩阵乘法循环交换优化实验

## 实验目的
验证循环嵌套顺序对程序性能的影响，直观感受**缓存未命中（cache miss）**如何导致执行时间剧增，为“算法-数据布局-硬件特性交互决定程序效率”这一观点提供实验依据。

## 实验环境
- 操作系统：Debian (WSL2 / Docker 容器)
- 编译器：Clang 14.0.0
- 编译选项：`-O2 -g`
- 矩阵大小：1024×1024 (double 类型)
- 计时方法：C 标准库 `clock()` 与 Shell `time` 命令

## 代码说明
`matmul_ijk.cpp` 与 `matmul_ikj.cpp` 实现了完全相同的矩阵乘法 C = A×B，唯一区别是循环嵌套顺序：
- **i-k-j 版本**：最内层循环对 `C[i][j]` 的访问是跨行跳跃的（步长 = N），缓存空间局部性极差。
- **i-j-k 版本**：最内层循环连续访问 `B` 的一列和 `C` 的一个元素，对缓存的利用效率高得多。

## 运行结果
执行脚本 `test.bash`，输出如下：
i-j-k: 0.467980 seconds
real 0m0.494s user 0m0.466s sys 0m0.028s

i-k-j: 0.761755 seconds
real 0m0.773s user 0m0.764s sys 0m0.008s

text

- `i-j-k`（连续访存）耗时约 0.47s（程序计时）或 0.49s（真实时间）。
- `i-k-j`（跳跃访存）耗时约 0.76s（程序计时）或 0.77s（真实时间）。  
- **性能差距：i-k-j 版本慢了约 63%**（用户态时间 `user` 从 0.466s 增至 0.764s）。

## 结论与分析
1. **循环交换产生了可测量的巨大性能差异**：仅仅交换两层循环的顺序，在同一台机器、同一编译器、同一优化级别下，执行时间相差近一倍。这直接验证了简历中的观点——“程序的计算效率并非单独由算法决定，而是由算法、数据结构布局、编译调度策略与硬件特性之间的复杂交互所决定”。
2. **缓存未命中是根本原因**：`i-k-j` 版本对矩阵 C 的访问模式是跨行跳跃的，每访问一个元素都需要加载新缓存行，导致 cache miss 激增。虽然本次未安装 `perf` 进行精确统计，但 `user` 时间的大幅增加正是 CPU 等待内存的典型表现。
3. **优化空间的结构性**：这个简单的 3 层循环嵌套其实就是一个微型的“优化空间”。不同的嵌套次序对应空间中的不同点，而性能的剧烈变化说明该空间存在“高速区域”（i-j-k）和“低速区域”（i-k-j）。若能识别这种结构，搜索算法便能避开低速区。这为“学习型编译器”中“用模型捕获优化空间结构”的想法提供了最基础的实验动机。
